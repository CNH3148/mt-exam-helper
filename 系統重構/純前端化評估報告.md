# 刷題系統：純前端化 (Pure Frontend / PWA) 評估與重構建議報告

## 1. 方案概述與願景
本方案旨在拔除現有的 Python 後端伺服器 (FastAPI)，將「刷題系統」完全轉型為純前端應用程式 (Pure Frontend) 或漸進式網頁應用程式 (PWA)。
* **核心願景**：實現真正的「零安裝、跨平台、跨裝置」。使用者只需開啟網頁連結（或開啟本地 HTML），即可在 Windows、Mac，甚至是 **iOS/Android 手機與平板** 上無縫刷題。

## 2. 現有架構痛點與挑戰
目前系統大量依賴 Python 後端做為「本地資料庫管理員」，直接讀寫使用者的本地硬碟檔案：
1. **讀取靜態資料**：載入 `data/` 內的 JSON 題庫。
2. **進度存檔管理**：直接建立、讀寫、刪除 `saves/` 目錄下的存檔 JSON。
3. **題庫修改與標籤**：修改答案、新增標籤、儲存搜尋條件時，後端會直接覆寫 `data/` 內的原始檔案。
4. **自訂圖片上傳**：允許使用者上傳圖片至 `public/images/`，並修改對應的 JSON。

若轉為純前端，瀏覽器基於安全性限制，**無法靜默讀寫本地硬碟檔案**，這是重構的最大障礙。

## 3. 實行建議與架構重構方針

為了解決上述挑戰，若要執行純前端化，建議採取以下重構方針：

### 3.1 資料儲存層轉移 (IndexedDB)
* **策略**：捨棄直接讀寫 `.json` 檔案的模式。前端應引入瀏覽器內建的 **IndexedDB**（建議搭配 `Dexie.js` 函式庫簡化操作）作為本地資料庫。
* **實作**：
  * **唯讀題庫**：將原始 `data/` JSON 視為「唯讀」的基礎資料。
  * **差異化儲存 (Delta Storage)**：使用者的存檔、新增的標籤、修改的答案等，全部存入 IndexedDB。
  * **資料合併**：前端載入題目時，即時將「基礎資料」與「IndexedDB 內的自訂修改」進行合併渲染。

### 3.2 解決 CORS 與跨裝置存取 (PWA 雲端託管)
* **問題**：若使用者直接雙擊本地的 `index.html`，瀏覽器會因 CORS 限制禁止 `fetch()` 讀取旁邊的 JSON 檔。
* **策略**：將整個純前端專案託管到免費的靜態網頁伺服器（如 GitHub Pages、Vercel）。
* **PWA 支援**：實作 Service Worker，將所有題庫與靜態資源快取到瀏覽器。這樣即使斷網，使用者依然可以開啟網頁離線刷題。

### 3.3 存檔攜帶性 (匯入/匯出機制)
* **問題**：IndexedDB 的資料綁定在單一瀏覽器。若使用者換電腦、換瀏覽器或清理快取，進度會消失。
* **策略**：在系統設定介面中，實作顯眼的「備份進度 (Export)」與「還原進度 (Import)」功能，讓使用者下載 `.json` 備份檔，手動實現跨裝置的進度轉移。

### 3.4 圖片上傳功能的妥協
* **問題**：純前端無法將圖片存入原本的 `public/images/` 資料夾。
* **策略 (二選一)**：
  1. **轉換編碼**：將上傳的圖片轉為 Base64 字串存入 IndexedDB。缺點是會大幅佔用瀏覽器儲存空間並影響效能。
  2. **改變機制**：移除本地圖片上傳功能，改為讓使用者「貼上網路圖片的 URL」來取代。

## 4. 總結
純前端化是一項大工程，需要重寫所有的資料存取邏輯。但只要完成這項重構，並將其部署為 PWA，本系統將能擺脫所有作業系統與環境的束縛，達成在手機端隨時隨地刷題的終極目標。
